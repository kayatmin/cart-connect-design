# CartConnect Mobile design.md

## Product

CartConnect Mobile is the agent-side Flutter app for live shopping support. Store agents use it to sign in, watch the live session queue, accept customer sessions, chat with shoppers, edit carts, add inventory items, and move sessions toward checkout.

## Memorable Thing

The first read should be: this is a calm command center for high-pressure commerce moments.

## Target User

Store support agents working on phones or small tablets while helping customers in real time. They need fast scanning, clear priority, big touch targets, and confidence that cart and chat changes landed.

## Project Type

Mobile operational app with live queue management, session detail, chat, cart editing, inventory add flow, auth recovery, empty states, and network recovery states.

## Aesthetic Direction

### Direction

Industrial/Utilitarian with premium retail restraint.

The app should feel like a compact operations console, not a consumer shopping app and not a dense enterprise dashboard. The visual system should be quiet, precise, and fast to parse under live support pressure.

### Decoration Level

Intentional.

Use tonal surfaces, small status rails, thin dividers only where they improve scan speed, and rare high-signal accent color. Avoid decorative gradients, bubbly shapes, centered marketing layouts, stock imagery, and heavy shadows.

### Mood

Calm, exact, awake. Waiting sessions are allowed to feel urgent, but the app itself should never feel frantic.

## Design Goals

- Mobile-first and practical for Flutter.
- Strong queue triage: waiting sessions are the most visible item on the dashboard.
- Clear selected-session continuity while the list refreshes.
- Chat and cart can be scanned without losing session state.
- Auth and network failures feel recoverable, not alarming.
- Visual system should map cleanly to current backend session, chat, cart, checkout, and inventory APIs.

## Core Screens

### 1. Splash / Bootstrap

- Brand mark or wordmark only.
- Small operational status line such as "Syncing agent workspace".
- No marketing copy.
- Background uses the base surface, not a decorative hero.

### 2. Login

- Strong CartConnect identity.
- Email and password fields.
- Primary sign-in CTA.
- Debug/API environment footer only for development builds.
- Trust posture: secure, restrained, work-focused.

### 3. Agent Dashboard

- Agent identity and availability summary.
- Compact KPI row for waiting, active, completed, and total.
- Session list grouped by status priority:
  - Waiting first, with amber left rail and stronger metadata.
  - Active second, with blue rail and clear "in progress" label.
  - Completed, expired, and declined last, in subdued neutral treatment.
- Preserve selected session during refresh.
- Empty queue state should feel good, not broken.

### 4. Session Detail

- Customer identity, status chip, and short session metadata at the top.
- Primary actions: accept, decline, end.
- Cart and chat both visible, with the active task taking more vertical space.
- Mutation state must be local to the selected session and action.
- Inline error callouts near the thing that failed.

### 5. Chat Focus

- Chat timeline as primary content.
- Customer messages left, agent messages right.
- Agent messages use the primary ink surface or blue-tinted container.
- Composer anchored to bottom with clear send affordance.
- Small customer/cart context chip stays visible without crowding messages.

### 6. Cart Edit

- Cart contents as primary content.
- Product image thumbnail, title, variant, price, quantity controls, remove action.
- Quantity controls must have 44px minimum hit area.
- Remove action is secondary and intentional, not visually loud.
- Subtotal and checkout readiness are sticky or visually anchored.
- Room for future discount and checkout actions.

### 7. Inventory Add Sheet

- Search field first.
- Results show product image, title, variant, availability, and price.
- Add action is explicit.
- Quantity stepper should not shift layout while loading.
- Loading and empty search states are quiet and useful.

### 8. Auth Expired

- Single-focus re-auth screen.
- Headline: "Session expired" or "Sign in again".
- Explain calmly that the agent needs to sign in again.
- One primary CTA.

### 9. Empty Queue

- "All clear" posture.
- Soft green or neutral success surface.
- No required action.
- Keep it compact enough that dashboard chrome remains recognizable.

### 10. API / Network Error

- Recoverable warning tone.
- Explain whether connection, server, or timeout when known.
- Primary retry button.
- Contact support fallback.
- Avoid technical jargon in user-facing copy.

## Color

### Approach

Restrained. One primary ink, one active blue, one waiting amber, semantic states, and layered neutrals.

### Palette

- Primary ink: `#102033` - headers, primary buttons, selected bottom nav, agent-owned message bubbles.
- Primary ink dark: `#07111F` - pressed state and high-emphasis text.
- Active blue: `#2563EB` - active session state, links, focused controls.
- Waiting amber: `#F2A900` - waiting sessions, queue urgency, time-sensitive badges.
- Clear green: `#167A55` - empty queue, success, completed session signals.
- Recoverable warning: `#8A5A00` - network retry warnings and non-destructive alerts.
- Danger: `#B42318` - destructive confirmation and failed mutation text only.
- Background: `#F6F7F9` - app base.
- Surface: `#FFFFFF` - cards and panels.
- Surface raised: `#FBFCFE` - inputs, chat composer, sticky control areas.
- Surface muted: `#EEF1F5` - inactive tabs, disabled controls, low-emphasis groups.
- Border soft: `#DCE2EA` - structural borders.
- Text primary: `#111827` - main copy.
- Text secondary: `#586174` - metadata and helper text.
- Text tertiary: `#8A94A6` - timestamps and disabled labels.

### Semantic Containers

- Waiting container: `#FFF4D6` with text `#6F4700`.
- Active container: `#E8F0FF` with text `#1D4ED8`.
- Success container: `#E7F6EE` with text `#166D4B`.
- Danger container: `#FDECEC` with text `#9F1D14`.
- Warning container: `#FFF7E8` with text `#8A5A00`.

### Dark Mode Strategy

Dark mode is not the first target. When added, redesign surfaces rather than inverting colors. Reduce accent saturation by 10-20 percent and keep waiting amber readable without making the whole app feel hot.

## Typography

### Typeface Direction

- Display / Screen titles: Plus Jakarta Sans, 700.
- Body and UI: Source Sans 3, 400-600.
- Data / timestamps / compact counts: IBM Plex Mono, 500, tabular numbers.
- Code / debug labels: IBM Plex Mono.

This avoids the generic Inter-style admin look while staying practical for Flutter and readable on small screens.

### Scale

- Display: 28px / 34px, weight 700.
- Screen title: 22px / 28px, weight 700.
- Section title: 17px / 24px, weight 700.
- Card title: 15px / 20px, weight 700.
- Body: 14px / 20px, weight 400.
- Body strong: 14px / 20px, weight 600.
- Label: 12px / 16px, weight 600.
- Metadata: 11px / 14px, weight 500.
- Data: 13px / 18px, weight 600, tabular numbers.

### Flutter Loading

Prefer bundled font assets for production. If using Google Fonts during exploration, map to:
- Plus Jakarta Sans for titles.
- Source Sans 3 for body/UI.
- IBM Plex Mono for data.

## Spacing

### Base Unit

4px.

### Density

Compact-comfortable. Agents need enough information on screen, but touch targets cannot feel cramped.

### Scale

- `2xs`: 2px
- `xs`: 4px
- `sm`: 8px
- `md`: 12px
- `lg`: 16px
- `xl`: 24px
- `2xl`: 32px
- `3xl`: 48px

### Touch Targets

- Minimum tap target: 44px.
- Primary button height: 52px.
- Text input height: 52px.
- Quantity stepper control: 40px visual height, 44px hit area.

## Layout

### Approach

Grid-disciplined for app screens, with small editorial offsets in headers and status summaries.

### Mobile Rules

- Page margin: 16px.
- Card padding: 16px.
- Dense list item padding: 12px vertical, 14px horizontal.
- Section gap: 20px.
- Bottom nav: fixed, 4 items max, selected item uses filled pill.
- Sticky composer and checkout actions use raised surface with top border.

### Responsive Rules

- Phones: single-column queue/detail flow with tabs or drill-in panels.
- Large phones / small tablets: queue and selected session may sit side by side.
- Desktop previews are secondary; mobile is the source of truth.

### Radius

- Tiny: 4px for rails and small indicators.
- Small: 8px for chips and compact controls.
- Medium: 12px for inputs and list rows.
- Large: 16px for cards and sheets.
- Extra large: 20px for major panels.
- Full: 999px for pills.

## Components

### App Scaffold

- Background `#F6F7F9`.
- Top app bar white or transparent depending on scroll.
- Avoid oversized hero headers inside the app.

### Status Rail

Use a 4px left rail on session rows:
- Waiting: amber.
- Active / joining: active blue.
- Completed: clear green.
- Expired / declined: neutral.

### Session Row

- Customer name or fallback ID.
- Status chip.
- Wait time or last activity.
- Cart subtotal and chat count.
- Selected state uses `#E8F0FF` background and primary ink text.

### KPI Tiles

- Compact row, not large cards.
- Label, count, and small status color.
- Waiting tile may use amber container.

### Chat Bubbles

- Customer: white card with soft border.
- Agent: primary ink bubble with white text.
- System: small centered neutral chip.
- Timestamps use tertiary text or mono data style.

### Cart Item Row

- 56px product thumbnail.
- Title, variant, price.
- Quantity stepper aligned right.
- Remove icon button muted by default, danger only on confirm.

### Buttons

- Primary: primary ink background, white text.
- Secondary: white background, soft border, primary ink text.
- Waiting action: amber container only when the action is truly time-sensitive.
- Danger: danger text or danger container, never as the default dominant action.

### Inputs

- Filled raised surface.
- Soft border only on focus or error.
- Labels stay outside the input when space allows.

### Error Callouts

- Use warning container for retryable network issues.
- Use danger container for failed destructive or auth actions.
- Keep copy short and action-oriented.

## Motion

### Approach

Minimal-functional.

### Rules

- State changes: 150-220ms ease-out.
- Bottom sheets: 220-300ms ease-out.
- Loading skeleton shimmer only where content shape is known.
- No looping decorative motion except a small live status pulse.
- Preserve scroll position and selected session across refresh.

## Stitch Generation Guidance

When generating screens from this file:
- Generate mobile-sized screens first.
- Use realistic agent support content, not lorem ipsum.
- Show live shopping support, queue triage, customer chat, collaborative cart editing, inventory add, and checkout readiness.
- Keep the same design system across all screens.
- Prefer production-looking layouts over wireframes.
- Avoid purple gradients, centered marketing layouts, generic feature cards, bubbly all-purpose radius, and stock-photo-heavy designs.
- Make waiting sessions visually louder than active and completed sessions.
- Keep auth and network recovery calm and single-purpose.
